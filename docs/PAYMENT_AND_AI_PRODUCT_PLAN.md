# Htex Visa 付费、模板与 AI 诊断产品方案

> 基于 2026-07-20 代码现状  
> 产品目标：两个付费触发入口，一个统一服务包，一次支付同时解锁正式模板与完整 AI 诊断。

## 1. 结论

推荐售卖的不是“去水印”或“一次 AI 调用”，而是按单次签证申请售卖的：

**完整申请服务包**

包含：

- 正式无水印申请模板；
- Word/PDF 下载；
- 完整材料风险诊断；
- 逐项修改建议；
- 材料与表单一致性检查；
- 有效期内重新生成模板；
- 有效期内重新诊断。

产品允许出现两个付费触发入口：

1. 用户生成带水印模板后，点击“解锁正式模板”；
2. 用户完成免费扫描后，点击“查看完整 AI 诊断”。

两个入口都创建或复用同一张待支付订单，支付成功后两项权益同时生效，不能让用户重复购买。

## 2. 代码现状

### 2.1 已经具备

- 前端已有完整材料向导：`MaterialWizard.vue`。
- 用户可在未登录状态填写，提交时才触发登录。
- 提交后会创建订单并跳转 `PaymentCheckout.vue`。
- 后端已有 `created → paid → completed` 主状态。
- Stripe/Mock 支付、Webhook、幂等、退款框架已经存在。
- 后端已有平台服务费配置：原价、促销价和促销窗口。
- 材料处理已有 OCR、分类、银行流水结构化分析。
- 材料诊断已有规则引擎与 RAG 参考。
- 前端已有模板预览组件 `MaterialTemplatePreview.vue`。

### 2.2 当前缺失

- 没有“购买权益”模型，系统只能通过订单状态猜测用户是否有权限。
- 模板是静态参考预览，不是基于订单生成的正式交付物。
- 没有正式模板文件、版本、下载权限和水印状态。
- `/materials/diagnose` 当前登录后即可获得完整问题、原因和修改建议，没有付费门槛。
- `/diagnose` 和 `/materials/diagnose` 都叫“诊断”，但产品定位不同，容易混淆。
- 支付页只显示订单号、金额和渠道，没有展示服务包内容。
- 支付成功后没有统一的“生成模板 + 深度诊断”处理任务。
- 诊断完成接口只更新状态，没有保存诊断报告。

### 2.3 必须先修正的金额问题

当前存在三套金额来源：

1. `PlatformPricing`：平台服务费，默认原价 USD 99.90、促销价 USD 19.90；
2. `VisaDestination.visa_fee_usd`：目的地费用；
3. `Order.total_amount`：创建订单时从目的地费用写入。

而支付页把 `order.total_amount` 直接四舍五入后当作 cents 使用，金额单位存在不一致风险。后端 Stripe 校验又会把 `total_amount × 100` 作为应付 cents。

产品和代码必须统一：

- `consular_fee`：使馆/签证中心费用，仅展示，不由平台代收；
- `service_fee`：Htex 完整申请服务包费用，平台实际收取；
- `order.total_amount`：始终保存服务费的主货币单位，例如 19.90 USD；
- 支付接口只接受后端根据订单价格快照计算出的金额，前端不得决定最终扣款金额。

## 3. 产品定位与命名

### 3.1 免费签证评估

对应现有 `/diagnose`。

用途：

- 获客；
- 用户尚未上传材料；
- 根据个人条件给出准备度评分；
- 免费展示完整结果；
- CTA 引导开始办理。

建议名称：

- “免费签证条件评估”
- 不使用“AI 材料诊断”名称。

它不产生付费权益，也不进入订单状态。

### 3.2 申请材料诊断

对应现有 `/materials/diagnose`。

用途：

- 针对实际上传材料；
- 检查缺件、格式、材料冲突、流水异常等；
- 免费展示摘要；
- 完整报告属于服务包权益。

建议名称：

- “申请材料体检”
- 付费结果名称：“完整 AI 材料诊断报告”。

## 4. 目标用户流程

### 4.1 主流程

```text
选择目的地和签证类型
→ 查看材料清单、使馆费与平台服务费
→ 免费填写资料
→ 免费上传/OCR/自动分类
→ 免费生成带水印模板预览
→ 免费完成基础材料扫描
→ 展示模板预览 + 诊断摘要
→ 用户从任一入口点击解锁
→ 登录（如未登录）
→ 创建待支付订单并保存价格快照
→ 支付完整申请服务包
→ 支付成功后上传/关联必要数据
→ 后台生成正式模板并运行深度诊断
→ 结果中心展示模板与诊断报告
→ 用户修改资料后重新生成/重新诊断
```

### 4.2 模板入口

```text
点击“一键生成模板”
→ 本地或服务端生成预览
→ 展示带水印版本
→ 点击“解锁无水印版本”
→ 权益检查
   ├─ 已购买：直接生成/下载正式版
   └─ 未购买：进入服务包确认页
```

### 4.3 AI 诊断入口

```text
点击“开始材料体检”
→ 免费规则扫描
→ 展示完整度、风险数量和风险分类
→ 点击“查看问题和修改建议”
→ 权益检查
   ├─ 已购买：返回完整报告
   └─ 未购买：进入服务包确认页
```

## 5. 免费与付费边界

| 能力 | 免费 | 付费 |
|---|---|---|
| 选择国家和签证类型 | 完整 | 完整 |
| 材料清单 | 完整 | 完整 |
| 信息填写与本地保存 | 完整 | 完整 |
| OCR 与材料分类 | 完整 | 完整 |
| 基础规则检查 | 缺件、清晰度、过期、格式 | 完整 |
| 模板 | 带水印/部分遮罩预览 | 无水印、编辑、下载、重生成 |
| 材料诊断 | 分数、等级、问题数量、分类 | 问题详情、证据、原因、修改建议 |
| 二次诊断 | 不开放 | 有效期和次数内开放 |

### 5.1 模板预览规则

- 水印使用“预览版本 · 不可用于提交 / PREVIEW · NOT FOR SUBMISSION”；
- 水印覆盖正文关键区域，不能只放页脚；
- 解释信、邀请函等纯文本内容需要隐藏关键段落，避免人工抄录；
- 多页文件默认展示第一页完整预览和其余页面缩略图；
- 免费预览禁止导出高清文件；
- 支付价值文案必须强调正式交付、诊断和持续修改，不能只写“去水印”。

### 5.2 诊断预览规则

付款前可以返回：

- `readiness_score`；
- `overall_risk`；
- 各严重等级问题数量；
- 问题类别；
- 已检查材料数；
- 已运行规则数。

付款前不得返回：

- 具体问题标题；
- 具体冲突字段；
- 证据原文；
- 完整问题详情；
- 可直接执行的修改建议。

## 6. 商品、订单与权益

### 6.1 MVP 商品

MVP 只保留一个 SKU：

```text
sku: VISA_APPLICATION_PACKAGE
scope: 单个用户 + 单个目的地 + 单个签证类型 + 单次申请
```

权益：

- `template_preview`
- `template_official`
- `template_download`
- `diagnosis_preview`
- `diagnosis_full`
- `diagnosis_rerun`

免费权益不需要购买；付费权益在支付回调成功后发放。

### 6.2 不建议直接用订单状态判断权益

订单状态描述业务处理进度，权益描述用户能访问什么。两者需要分开。

建议新增：

#### `products`

- `sku`
- `name_i18n_key`
- `active`
- `entitlements_json`

#### `order_price_snapshot`

可以先作为 orders 字段或独立表：

- `product_sku`
- `list_price_cents`
- `sale_price_cents`
- `currency`
- `pricing_version`
- `promotion_id`
- `consular_fee_display_json`

#### `order_entitlements`

- `order_id`
- `user_id`
- `entitlement_code`
- `status`: active/revoked/expired
- `granted_at`
- `expires_at`
- 唯一键：`order_id + entitlement_code`

支付 Webhook 幂等发放，退款是否撤销权益按退款政策执行。

### 6.3 订单状态

保留现有主状态，但补充处理状态：

```text
created
→ paid
→ processing
→ completed

created → cancelled
paid/processing/completed + refund_status 副轨
```

如果不想增加 `processing` 主状态，可以增加：

- `fulfillment_status`: pending/processing/ready/partial_failed/failed

推荐使用独立 `fulfillment_status`，避免破坏已上线订单状态兼容性。

## 7. 模板系统方案

### 7.1 当前组件的定位调整

`MaterialTemplatePreview.vue` 保留为“参考样式和预览渲染器”，但不应直接承担正式文件交付。

新增正式模板领域：

#### `generated_documents`

- `id`
- `order_id`
- `template_code`
- `template_version`
- `language`
- `status`: queued/generating/ready/failed
- `preview_storage_key`
- `official_storage_key`
- `watermarked`
- `source_fingerprint`
- `created_at`
- `expires_at`

### 7.2 模板生成接口

- `POST /orders/{order_no}/documents/preview`
  - 免费；
  - 返回带水印预览或预览任务；
  - 不要求付费权益。

- `POST /orders/{order_no}/documents/generate`
  - 要求 `template_official` 权益；
  - 生成正式 Word/PDF。

- `GET /orders/{order_no}/documents`
  - 返回模板列表与生成状态。

- `GET /orders/{order_no}/documents/{id}/download`
  - 要求 `template_download` 权益；
  - 返回短效签名下载地址。

### 7.3 重生成规则

- 资料没变化时重复点击，复用已有文件；
- 资料变化后计算 `source_fingerprint`；
- 指纹变化才重新生成正式文件；
- 新文件生成成功后替换当前正式版本，不向用户提供历史版本恢复；
- 服务有效期内的当前正式结果持续可访问；
- 单订单设置合理频率限制，防止无限生成。

## 8. AI 诊断系统方案

### 8.1 两阶段诊断

#### Preview

使用现有 `VisaDiagnoser` 的确定性规则先跑，返回脱敏摘要。

#### Full

支付后：

- 复用规则结果；
- 加入银行流水、跨材料一致性和语义分析；
- 保存结构化诊断报告；
- 返回完整问题与建议。

### 8.2 诊断报告持久化

新增 `diagnosis_reports`：

- `id`
- `order_id`
- `version`
- `status`: queued/running/ready/partial_failed/failed
- `input_fingerprint`
- `overall_risk`
- `risk_score`
- `summary`
- `issues_json`
- `positives_json`
- `policy_refs_json`
- `rule_version`
- `model_provider`
- `model_name`
- `created_at`
- `completed_at`

必须保存报告，不能像当前接口一样只返回一次结果后丢失。

### 8.3 诊断接口

- `POST /orders/{order_no}/diagnosis/preview`
  - 登录；
  - 免费；
  - 返回摘要。

- `POST /orders/{order_no}/diagnosis/run`
  - 要求 `diagnosis_full` 权益；
  - 创建异步任务；
  - 幂等复用相同输入指纹的报告。

- `GET /orders/{order_no}/diagnosis/latest`
  - 未购买只返回 preview DTO；
  - 已购买返回 full DTO；
  - 权限在后端裁剪，不能只靠前端隐藏。

- `POST /orders/{order_no}/diagnosis/rerun`
  - 要求 `diagnosis_rerun` 权益；
  - 校验次数、时间和输入是否变化。

### 8.4 AI 失败策略

- 规则引擎成功、LLM 失败：报告状态 `partial_failed`，先交付规则结果；
- LLM 超时不影响支付成功；
- 页面提示“部分高级分析正在重试”，不能把订单直接标记失败；
- 后台支持重试与人工查看错误原因；
- 禁止使用“保证出签”“官方结论”等表述。

## 8.5 资料修改、版本与插件授权码

### 当前代码现状

- 材料向导只在 `localStorage` 保存一份最新草稿；
- 草稿按国家和签证类型区分，保存期限为 7 天；
- 每次编辑会覆盖上一次草稿，没有历史版本和恢复能力；
- 正式模板没有版本表；
- AI 诊断报告没有持久化版本；
- 支付后的 `applicant_data` 也是订单上的单份最新数据。

所以当前不能向用户承诺“保留 3 个版本”或“可以恢复旧版本”。

### 推荐的付费版本政策

完整申请服务包建议明确为：

- 服务有效期：支付后 365 天；
- 用户可不限次数修改表单草稿；
- 正式模板最多重新生成 20 次；
- 完整 AI 诊断最多重新运行 10 次；
- 系统只维护当前正式版本，不向用户提供历史版本查看或恢复；
- 新结果生成成功后替换当前结果；
- 纯预览不消耗正式生成次数；
- 因系统失败产生的重试不计入次数。

这里需要区分：

- “修改次数”：表单字段可以不限次数修改；
- “正式生成次数”：会消耗模板或 AI 成本，按套餐限制；
- “当前版本”：用户始终访问最近一次成功生成的正式结果，没有历史恢复入口。

### 推荐的修改流程

```text
订单工作台
→ 点击“修改申请资料”
→ 基于当前资料创建可编辑草稿
→ 用户修改字段或替换材料
→ 页面展示变更摘要
→ 点击“保存修改”
→ 生成新的资料 fingerprint
→ 在重新生成完成前，旧正式结果暂时仍可查看
→ 用户选择：
   ├─ 重新生成正式模板
   ├─ 重新运行 AI 诊断
   └─ 暂不更新结果
→ 新结果成功后替换当前正式结果
```

保存字段修改本身不应该自动消耗生成次数。只有用户确认“重新生成”或“重新诊断”时才扣次数。

### 插件 12 位授权码的变化

当前插件使用的不是使馆网站验证码，而是 Htex 自己签发的 12 位资料授权码。

现有代码规则：

- 仅已支付、可用状态的订单可以生成；
- 授权码有效期为 10 分钟；
- 授权码只能成功兑换一次；
- 每次重新签发都会撤销之前的授权码；
- 申请资料变化会改变资料 fingerprint；
- 资料变化后旧授权码兑换会返回 `DS160_ARCHIVE_CHANGED`；
- `set_applicant_data` 更新申请资料时也会主动撤销已有授权码；
- 插件成功兑换后，资料保存在 `chrome.storage.session`，关闭浏览器后清除。

推荐产品流程：

```text
用户修改资料
→ 保存为新的当前资料
→ 系统立即撤销尚未兑换的旧授权码
→ 工作台提示“资料已更新，请生成新的插件授权码”
→ 用户点击“生成新授权码”
→ 插件兑换新码
→ 插件清除旧会话资料并载入当前资料
→ 用户重新执行填表或仅更新变化字段
```

需要注意：插件已经成功兑换并保存在当前浏览器会话中的旧资料，不会因为网站端资料修改而自动消失。为避免误填，插件在开始填表前应调用轻量版本校验接口：

- 当前 fingerprint 与服务器一致：允许继续；
- fingerprint 已变化：阻止继续填表，清除旧会话资料，并提示重新兑换；
- 无网络：明确提示无法确认资料是否为最新版，不建议继续自动填表。

建议新增：

- `GET /api/v2/ds160/archive-version/{order_id}`
  - 只返回 fingerprint 前缀、版本号和更新时间；
  - 不返回任何申请人资料；
  - 插件填表前和页面切换时调用。

使馆网站自己的图片验证码、短信验证码或会话验证码不属于 Htex 12 位授权码。它们由使馆网站控制，可能随页面刷新、会话超时或提交步骤变化，Htex 不能承诺保持不变，也不应把两者合并描述。

## 9. 支付页改版

当前支付页应从“订单号 + 金额 + 渠道”升级为“服务包确认页 + 支付组件”。

页面内容：

1. 标题：解锁完整申请服务；
2. 用户当前价值摘要：
   - 已生成几份模板预览；
   - 已扫描几份材料；
   - 已发现几类问题；
3. 服务包权益列表；
4. 原价、促销价、币种；
5. 明确说明使馆费不包含在本次付款内；
6. 退款规则；
7. 支付按钮；
8. 安全支付和隐私说明。

入口参数可以记录 `source=template|diagnosis`，仅用于差异化标题与漏斗统计，不影响商品和价格。

## 10. 支付成功后的结果中心

不建议支付后直接进入普通订单详情。建议新增：

`/orders/{orderNo}/workspace`

结构：

- 顶部：订单、目的地、服务包状态；
- Tab 1：材料准备度；
- Tab 2：完整 AI 诊断；
- Tab 3：正式模板；
- Tab 4：下一步办理指引。

处理状态：

```text
支付成功
→ 正在生成正式模板
→ 正在进行完整材料诊断
→ 部分结果已完成
→ 全部结果可用
```

模板和诊断独立失败、独立重试，不能因为其中一个失败阻塞另一个结果。

## 11. 隐私设计

现有代码采用“付款前材料和申请数据主要保存在浏览器，付款后才上传申请人数据”的思路，应继续保留。

### 11.1 对外承诺必须与技术实现一致

产品希望在支付时提示“资料仅保存在本地，不会上云”，这个方向可以成为核心信任卖点，但当前代码还不能直接使用该承诺：

- `PaymentCheckout.vue` 在支付成功后调用 `syncPendingApplicantAfterPayment`；
- 后端 `PUT /orders/{order_no}/applicant-data` 会把完整申请资料写入数据库；
- 正式模板若由后端生成，需要把模板所需字段传给后端；
- 完整 AI 诊断若调用外部模型服务商，材料内容可能离开用户设备或自有服务器。

因此，在完成对应技术改造前，禁止使用：

- “绝对安全”；
- “100% 安全”；
- “所有数据永不上云”；
- “任何情况下都不会上传”。

这类绝对化承诺无法覆盖浏览器同步、支付日志、错误日志、备份、模型调用和用户主动联系客服等场景，也会扩大合规风险。

### 11.2 推荐的数据模式

#### 模式 A：真正的本地处理模式（长期推荐）

- 原始护照、银行流水、证件图片只保存在用户浏览器或设备；
- OCR、基础诊断、水印模板和正式模板尽可能在浏览器本地生成；
- 服务器只保存订单号、价格、支付状态、权益状态和不含个人资料的履约状态；
- 如需云端 AI，先在本地提取并脱敏，仅上传诊断必需的结构化字段；
- 用户主动选择“使用云端高级诊断”时，再进行单独、明确的知情同意；
- 支付成功不再自动调用 `syncPendingApplicantAfterPayment` 上传完整申请资料。

这个模式完成后，可以使用“默认仅在本地处理”的强隐私表达。

#### 模式 B：自有服务器加密处理模式（MVP 更容易实现）

- 付款前资料主要保存在用户设备；
- 付款后，经用户明确同意，将生成正式结果所需资料加密传输至自有服务器；
- 服务端加密存储并设置自动删除期限；
- 第三方模型只接收最小化、脱敏后的必要内容；
- 页面明确列出处理目的、保存位置、保存期限和删除方式。

这个模式不能宣称“不会上云”，应表述为“不会在未经授权的情况下上传原始资料”。

### 11.3 支付页推荐文案

如果采用模式 A：

> **你的资料默认留在本机**  
> 护照、银行流水和申请材料默认仅在当前设备中处理。支付只用于开通正式模板和完整诊断权益，不会自动上传你的原始文件。若某项高级功能需要云端处理，我们会在上传前单独说明并征得你的同意。

如果采用模式 B：

> **我们尽量减少资料上传**  
> 付款前，你的申请资料主要保存在当前设备。支付后，仅在你确认后，才会加密上传生成正式模板和诊断所必需的信息。资料仅用于本次申请，并按照隐私政策在约定期限内删除。

支付按钮附近增加：

- `了解资料如何处理`
- `查看隐私政策`
- 一个不得默认勾选的确认项：
  - “我已了解本次服务的数据处理方式，并同意按照隐私政策处理必要信息。”

隐私说明不能藏在支付按钮下方的小号灰字中，也不能使用“继续支付即代表同意”代替明确确认。

### 11.4 数据处理说明弹窗

支付页的“了解资料如何处理”应打开简明弹窗，至少回答：

| 问题 | 必须说明 |
|---|---|
| 哪些数据留在本地 | 原始文件、OCR 结果、表单草稿等的实际范围 |
| 哪些数据会上传 | 订单、支付、必要结构化字段或原始材料 |
| 为什么上传 | 支付、生成正式模板、完整诊断或客服处理 |
| 上传到哪里 | 自有服务器及可能使用的第三方处理方 |
| 保存多久 | 明确期限，不能只写“必要期间” |
| 如何删除 | 用户入口、客服渠道和处理时限 |
| AI 如何使用 | 是否脱敏、是否用于训练、是否跨境 |

需要处理的关键场景：

- 用户在设备 A 生成预览，在设备 B 支付：支付后无法自动获得设备 A 的本地材料；
- 浏览器清缓存后，本地预览和诊断输入会丢失；
- 支付成功后 `syncPendingApplicantAfterPayment` 失败时，目前只写 console warning，用户可能已付款但无法生成结果。

改造要求：

- 支付前明确提示“支付完成后需要安全上传本次申请资料用于生成正式结果”；
- 支付成功后的资料同步必须有可见状态、重试按钮和错误恢复；
- 未同步成功不能把履约标记为 completed；
- 上传前获得隐私同意；
- 对模型服务商的数据处理、保存时间和跨境传输单独披露；
- 下载使用短效签名 URL；
- 日志不记录原始护照号、银行流水和完整提示词。

## 13. MVP 开发优先级

### P0：上线支付墙前必须完成

1. 统一服务费、使馆费和订单金额；
2. 后端生成订单价格快照；
3. 修正前端 cents/主货币单位；
4. 新增订单权益并由支付回调幂等发放；
5. 把材料诊断拆成 preview/full DTO；
6. 后端强制 full 结果权益校验；
7. 支付页展示服务包权益和“不含使馆费”；
8. 支付成功数据同步可重试；
9. 支付入口去重，已购买用户不再进入付款页。

### P1：形成完整商品体验

1. 动态水印模板预览；
2. 正式 Word/PDF 生成；
3. 正式文件持久化与短效下载；
4. 诊断报告持久化；
5. 订单工作台；
6. 模板和诊断异步履约状态；
7. 退款与权益撤销策略；
8. 管理后台查看履约状态。

### P2：优化转化与成本

1. 重新生成/重新诊断次数配置；
2. 按国家和签证类型配置不同模板；
3. AI Token 成本与失败率看板；
4. 支付入口 A/B 测试；
5. 套餐升级或人工服务增购；
6. 多设备草稿恢复。

## 14. 验收标准

### 商业逻辑

- 用户从模板或诊断入口进入，看到的是同一服务包；
- 同一申请只需支付一次；
- 支付后两个付费功能立即获得权限；
- 已购买用户不会再次出现支付按钮；
- 使馆费与平台服务费明确分开。

### 权限

- 未购买用户无法通过直接调用 API 获得完整诊断；
- 未购买用户无法下载无水印文件；
- 修改前端状态或 URL 不能绕过后端权益校验；
- 支付回调重复投递不会重复发放权益。

### 履约

- 支付成功后数据同步失败可恢复；
- 模板失败不影响诊断结果；
- AI 失败可以降级到规则报告；
- 相同输入不会重复消耗生成和模型成本；
- 完整报告和正式文件可以从订单工作台再次访问。

### 金额

- 定价页、服务包确认页、PaymentIntent 和订单记录金额完全一致；
- 所有支付金额由后端计算；
- 前后端统一使用 cents 或明确的 Decimal 转换；
- 促销结束后旧订单仍按价格快照支付，新订单使用新价格。

## 15. 推荐实施决策

本轮产品决策建议直接确定为：

1. 一个商品：完整申请服务包；
2. 两个付费入口：正式模板、完整 AI 诊断；
3. 免费模板可预览但带水印和内容保护；
4. 免费诊断只展示摘要；
5. 支付后两个权益同时解锁；
6. 使馆费不代收，只展示；
7. 服务费以订单价格快照为准；
8. 后端权益校验是唯一权限依据；
9. 支付后进入统一申请工作台；
10. 先完成 P0，再上线真实收费。
11. 隐私承诺采用“默认本地处理 + 云端功能单独授权”，不使用无法验证的绝对安全表述。
