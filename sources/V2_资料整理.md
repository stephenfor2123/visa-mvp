### A.2 V0 11 章核心要点(每章 3-5 条)

#### 第一章 全局业务规则 & 硬性约束
- **业务约束**:仅旅游/学生签、禁 B 端、用户自负真实性、禁违规词("包过/100%/官方授权/使馆合作")
- **技术&部署**:100% 本地部署、敏感数据非对称加密、临时文件 24h 删除/归档 180 天删除、仅 RPA 可外联、RPA 强制限流、**禁活体/指纹**
- **合规约束**:三语种协议、日志保留 1 年不可删、注销后 72h 销毁数据
- **V2 升级点**:明确 "纯工具版" 不涉及拒签险/保险打包/拒签退费(与 V1 调研报告的"拒签险套利"差异化)

#### 第二章 渠道 & 终端要求
- **支持终端**:App(Android/iOS) + Web(Chrome/Edge) + 小程序,三端数据互通
- **语种**:英语/印尼语/越南语,一键切换,所有文案/弹窗/协议多语种
- **V2 升级点**:Web 端需**自适配 PC 桌面**(V1 原型是 PC 设计稿)与移动浏览器;小程序作为轻量入口

#### 第三章 前端应用层
- **3.1 账号**:手机号注册(各国格式)+ 短信验证码、密码 6-16 位字母+数字、首次注册强制同意协议、记住登录 7 天、个人中心(订单/材料仓库/语种/注销)
- **3.2 首页/国家选择**:后台配置国家列表,前端固定;无自定义国家入口
- **3.3 材料采集(核心 1)**:OCR 拍照(边框辅助对齐) + 文件上传(JPG/PNG/PDF ≤10MB) + 语音录入(转字段),混合使用
- **3.4 AI 校验(核心 2)**:证件有效期、号码位数、图片质量(模糊/反光/遮挡)、基础格式四类规则,通过/警告/拒绝三档
- **3.5 RPA 填表(核心 3)**:自动加载官方表单 + 自动回填 + 用户编辑 + 验证码处理(图形/滑动,**无活体**) + 提交进度
- **3.6 订单状态**:5 态(待提交/已提交/审核中/出签/拒签) + 状态推送 + 超时标记
- **3.7 增值跳转**:拒签险/机票/酒店**外链**(不做深度接口)
- **V2 升级点**:V1 原型拆得更细 → V2 应保留扫描引导、AI 校验结果详情、RPA 提交进度、订单详情 4 个**新页面**

#### 第四章 业务中台(后端)
- **4.1 用户服务**:注册/登录/注销 + 权限(C 端) + 账号-订单-材料一对一绑定 + 注销 72h 清空
- **4.2 订单服务**:生命周期管理 + 全局唯一订单号 + 状态定时拉取(可配轮询周期) + 异常订单统计
- **4.3 文件服务**:接收/转码/缩略图 + 按订单/用户分目录 + 加密存储 + 自动清理(24h/180d) + **复用支持**
- **4.4 配置服务**:国家/AI 规则/RPA 策略/语种资源 4 类配置可视化
- **4.5 权限&后台**:管理员独立入口 + 超级管理员/普通运维两级 + 查看用户/订单/日志 + 修改配置
- **4.6 支付服务**:本地主流渠道 + 订单支付/退费 + 流水存档
- **4.7 日志&监控**:全链路日志 + 1 年保留 + 基础告警(宕机/RPA 失败/存储)
- **V2 升级点**:V0 4.5 提了"内部管理员后台",V2 纯工具版**仅保留"配置后台"**给开发者用(无运营角色),简化权限模型

#### 第五章 AI & RPA 专项
- **5.1 OCR & 语音**:证件/PDF/图文识别 + 印尼/越南/英语字符 + 倾斜/反光容错 + 字段映射回填
- **5.2 AI 校验引擎**:规则可配置(后台开关) + 三档结果(通过/警告/拒绝) + 多语种整改文案
- **5.3 RPA 机器人**:模拟人工访问 + 填表/提交 + **图形/滑动验证码解算** + 单 IP/单账号/单日/错峰 4 重限流 + 改版/失败自动告警 + 状态实时监控
- **V2 升级点**:V0 第 5.3 已覆盖 RPA 限流;V2 应**细化验证码识别引擎选型**和**多国模板切换**

#### 第六章 基础设施 & 本地部署
- **6.1 硬件**:应用服务器 + 数据库服务器(独立 MySQL/PG) + 存储服务器 + 备用容灾
- **6.2 软件**:Linux + 关系型 DB + Redis + 本地对象存储 + 消息队列
- **6.3 安全**:防火墙白名单 + 字段加密 + 文件加密 + 强密码 + 导出日志
- **6.4 备份**:DB 每日全量 + 文件每周增量 + 单独存放轮转
- **V2 升级点**:V0 6.1 提到 MySQL/PostgreSQL 二选一;V2 **§C 必须给出明确推荐**;V0 6.2 提到消息队列,V2 应明确**Bull / RabbitMQ 选型**

#### 第七章 端到端标准业务流程
- V0 已给出 12 步时序:登录 → 选国家 → 采集 → 校验 → 加载表单 → 编辑 → RPA 提交 → 订单生成 → 状态拉取 → 增值跳转 → 完结/退费 → 材料复用
- **V2 升级点**:V2 纯工具版**删除"拒签退费"环节**;新增"AI 校验失败整改闭环"细化(校验 → 红字标注 → 用户改 → 重校验,直到通过)

#### 第八章 非功能性需求(NFR)
- **性能**:页面加载(移动 ≤2s/Web ≤1.5s)、OCR(单图 ≤3s/PDF ≤8s)、AI 校验 ≤2s、RPA 提交流 ≤15s、并发 500 在线/100 单
- **兼容**:Android 8+ / iOS 14+ / Chrome+Edge / 主流图片/PDF 格式
- **可用**:整体 ≥99% + 定时维护窗口
- **安全**:无明文 + 防刷 + 强密码 + 日志不可篡改
- **V2 升级点**:V0 第 8 章偏简略,V2 应**补充可观测性指标**(OCR 准确率、提交成功率、验证码识别率、人工干预率)

#### 第九章 接口清单
- **内部**:用户/订单/文件/OCR-AI/RPA/配置/日志
- **外部**:短信(本地)、支付(本地)、签证官网(RPA)、外链跳转(拒签险/OTA)
- **V2 升级点**:V0 第 9 章仅给了接口名,**V2 必须补 RESTful 路径、方法、请求/响应 Schema**(T2 写作时需展开)

#### 第十章 验收标准
- V0 给出 6 类验收(账号/材料/RPA/后台/部署/合规),每类 3-4 条
- **V2 升级点**:V0 验收**没有量化指标**,V2 必须补:
  - OCR 字段准确率 ≥ 95%(中英文)、≥ 90%(印尼/越南文)
  - AI 校验规则覆盖率 100%(4 类规则全覆盖)
  - RPA 表单填充成功率 ≥ 95%、验证码识别率 ≥ 90%
  - 订单状态轮询 SLA ≤ 30 min
  - 三语种覆盖率 100%、翻译准确率(关键字段)

#### 第十一章 人员分工
- V0 列出 5 角色:产品 / 前端 / 后端 / AI&RPA / BD
- V0 第 11 章被截断,但已包含 5 个角色

## §C 技术选型建议(8 个关键决策)

> **V2 上下文**:8 周 MVP / 纯开发团队 / 纯工具版 / 本地部署 / 三端(Web+App+小程序) / 三语种 / 印尼+越南+菲律宾 C 端

| # | 决策项 | 推荐 | 一句话理由 | 备选与权衡 |
|---|---|---|---|---|
| 1 | **数据库** | **PostgreSQL 15** | 强类型 + JSONB 半结构化字段(适配签证表单) + 免费开源 + 团队熟悉 | MySQL 8 也可,但 JSON 支持弱;SQLite 仅适合单机 demo |
| 2 | **ORM** | **Prisma(前端) / SQLAlchemy(后端)** | Prisma 配 NestJS/Node 体验最佳,SQLAlchemy 配 FastAPI 体验最佳 | GORM 配 Go,但团队栈偏 JS/Python |
| 3 | **OCR 引擎** | **PaddleOCR(主) + Tesseract(兜底)** | PaddleOCR 中文/英文/拉丁文精度高、本地部署、轻量(<200MB);Tesseract 全语言兜底 | EasyOCR 部署简单但慢;云 OCR 违反本地部署约束 |
| 4 | **语音识别** | **FunASR(主) / Whisper.cpp(备)** | FunASR 中文/英文/东南亚语种好、CPU 可跑、阿里开源(纯本地);Whisper 兜底覆盖小语种 | Whisper 原版需 GPU,Whisper.cpp 可 CPU 但稍慢 |
| 5 | **RPA 框架** | **Playwright(主) + Camoufox(反检测)** | Playwright 异步 API 干净、跨浏览器、本地部署、Modern 反检测生态;Camoufox 是 Firefox 硬指纹混淆版 | Selenium 慢且老;Puppeteer 偏 Node,Python 绑定弱 |
| 6 | **后端框架** | **FastAPI(Python)** | 异步性能好 / 自动 OpenAPI 文档 / 与 AI&RPA(PaddleOCR/Playwright)同语言生态 / 三语种 i18n 简单 | NestJS 偏重企业,8 周 MVP 不必;Spring Boot 团队不熟 |
| 7 | **前端** | **Flutter(App+小程序 WebView) + 独立 Web(React/Vue)** | Flutter 一套代码出 iOS/Android/小程序(uni-app 转译),共享 80% 业务逻辑;Web 独立维护 SEO 友好 | RN 包体大、性能弱;三端独立 = 8 周做不完 |
| 8 | **验证码识别** | **ddddocr(主) + CNN 自训(备)** | ddddocr 国内开源、离线、识别简单图形/数字/字母准确率 >90%;CNN 自训覆盖滑动/复杂验证码 | 第三方 API(2Captcha)违反本地部署 + 贵 |

### C.1 选型附加说明

**数据库 + ORM 组合细节**:
- PostgreSQL 15 主库 + Redis(状态缓存/会话)+ 本地 MinIO(对象存储替代 S3)
- Prisma(若选 NestJS)或 SQLAlchemy + Alembic(若选 FastAPI)做迁移
- 7 张主表:user / order / material / country_template / rpa_task / payment_log / audit_log

**OCR/语音部署细节**:
- PaddleOCR 用 `paddleocr==2.7+` 推理,启用 `det+rec+cls` 全流程
- FunASR 用 `paraformer-zh` 英文/中文 + `SenseVoiceSmall` 多语种小模型(<500MB)
- 都跑在 CPU 模式,8 核 16G 起步,MVP 单机即可

**RPA 限流与风控**:
- Playwright + Camoufox(指纹混淆 Firefox)做反检测
- 单 IP 单日最大访问 ≤ 50 次,单账号提交间隔 ≥ 60s
- 错峰时段:UTC 0:00-6:00 集中提交(避开各国上班时间)
- 验证码连续失败 3 次自动终止任务,人工介入提示

**三端架构细节**:
- App:Flutter 3.x(单代码出 iOS/Android)
- 小程序:uni-app x 把 Flutter 业务逻辑编译成微信小程序(WebView 壳)
- Web:独立 React 18 + Vite + Tailwind(SEO 友好,营销页 + 桌面用户)
